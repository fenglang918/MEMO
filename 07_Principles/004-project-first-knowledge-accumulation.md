# Project-First Knowledge Accumulation

- Date: 2026-01-01
- Status: `#status/active`

- Keywords: 先项目后资产；知识抽象；避免伪工作；上下文优先
- Tags: `#area/knowledge-management` `#scene/learning` `#trigger/reading-notes` `#trigger/summarizing` `#scope/memo` `#status/active`
- Trigger signals:
  - 读完一篇文章就想单独写一份系统综述
  - 学了一个新工具就想立刻在资产区建长篇笔记
  - 感觉“某个概念很重要”，但当前没有具体项目要用它
- Decision rule:
  - **默认先把新知识放进当前正在解决的具体项目语境里。**
  - 只有在它已经被反复使用，或项目结束后确认具有稳定复用价值时，再抽象成 `04_Assets/` 里的通用资产。

## Context

- 脱离语境的“总结”很容易沦为伪工作：写得很完整，但之后再也不会回看。
- 在项目语境里沉淀的知识，即使更碎片化，也更容易被立即消费和检验。
- 先消费、再抽象，能显著提高知识的存活率和复用价值。

## Examples

- 正例：
  - 你在 `03_Goal-Projects/2026/01/MEMO-开源模板发布/` 中研究模板目录演进，就把思考先写进该项目的 `cognition/`，等模式稳定后再整理成通用 README 或 schema。
- 正例：
  - 你在某个长期项目里学习新的工作流脚本，先在项目文档中记录“这次为什么需要它、怎么用”，而不是立刻做一份脱离上下文的全局 cheatsheet。

## Anti-examples (Optional)

- 刚看完一个教程，还没有在任何项目中使用，就在 `04_Assets/` 里写一份大而全的教程摘抄。
