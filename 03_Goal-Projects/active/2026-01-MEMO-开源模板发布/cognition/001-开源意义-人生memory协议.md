# Cognition 001：开源意义——人生 memory 的协议与参考实现

## 背景

我现在在考虑做一个 “MEMO 仓库的开源模板” 来传播：我们既没有自动化数据源，也无法（或不想）做成直接提供 LLM API 的产品，那开源到底在交付什么价值？

## 当前观点 / 假设

- 我们在做的是“人生 memory 的协议（protocol）”：先把 outputs 定义清楚（对象模型 + 最小字段 + 可检索约定），再允许 inputs 无限多样（聊天截图/报告/网页/随手笔记）。
- “AI” 在这个体系里更像可插拔的编译器：用户用自己的信息源 + 自己选择的 AI 工具，把碎片编译进协议化输出；我们不提供托管服务，也不绑定某一家模型/平台。
- 开源交付的不是“数据”和“自动化”，而是“长期可迁移的管理方法”：让用户把记忆资产保持在本地文件/版本控制中，并在不同工具之间复用。

## 稳定 outputs（建议对象）

- People：人卡（关系、资源、下次跟进）
- Projects：目标/计划/认知/决策（Target/Plan/Cognition/Decision）
- Signals：偏好/约束/策略变化（可复用的自我信息）
- Index：tags/关键词/链接规则（保证检索稳定）

## 非目标（避免误解）

- 不做全自动采集（没有数据源整合、没有账号授权抓取）
- 不做产品化 CRM/社交漏斗（邮件/提醒/图谱/可视化先不做）
- 不做“记录一切对话”的仓库（只沉淀对决策/行动有用的结论）

## 证据与依据

- 现实约束：不同人的数据源碎片化、隐私边界差异巨大；强行自动化会带来平台锁定与合规风险。
- 结构优势：协议化 outputs 后，工具选择变成“可替换的实现细节”，而不是“记忆资产的唯一载体”。

## 风险与应对

- 风险：用户误以为这是“自动化产品” → 应对：README 明确边界；强调 search-first + manual ingest 的最小闭环。
- 风险：协议设计过重 → 应对：最小字段优先；可选字段逐步演进；允许“先能搜到人/事，再慢慢补全”。

## 未解问题（下一步要验证）

- [ ] Signals 是否需要单独一级对象，还是先作为 cognition/notes 的一类标签？
- [ ] Events 是否要单独建模（时间线），还是先通过项目与流水日志自然形成？
- [ ] 公开版需要到什么程度的“参考实现”（CLI/校验/示例 walkthrough）才能让新人真正上手？
